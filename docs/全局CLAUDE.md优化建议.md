```markdown

**禁止行为**：
```

## 三、严重问题

### ❌ 问题1: 缺少"上下文断裂应对策略"

**问题**: 今天15次上下文断裂，但全局CLAUDE.md没有任何关于"如何应对上下文断裂"的指导

**影响**:
- AI不知道上下文断裂后应该做什么
- AI不知道应该优先恢复哪些信息
- AI不知道应该问用户什么问题来快速恢复上下文

**建议**: 增加"上下文断裂应对协议"

```markdown
## 上下文断裂应对协议

### 当看到"This session is being continued from a previous conversation"时

**第一步：快速定位当前状态**
1. 检查项目CLAUDE.md的"当前状态"部分（如果有）
2. 检查最近的git commit message（最近5条）
3. 检查是否有未完成的TODO或已知问题

**第二步：识别关键遗漏信息**
优先恢复以下信息（按重要性排序）：
1. 🔴 **时间压力**: 用户是否处于紧急状态？（deadline、比赛、生产事故）
2. 🔴 **当前blockers**: 什么问题阻塞了进展？
3. 🟡 **临时约束**: 今天有什么特殊的约束或偏好？
4. 🟡 **已废弃方案**: 哪些方案已经尝试过但失败了？
5. 🟢 **技术决策背景**: 为什么选择当前的技术方案？

**第三步：主动询问（如果summary不清楚）**
用一句话询问，不要长篇大论：
- "我看到上下文断裂了。当前最紧急的问题是什么？"
- "继续之前的工作，还是有新的优先级？"

**禁止行为**：
- ❌ 不要重新讨论已经决定的技术选型
- ❌ 不要重复建议summary中明确提到"已尝试但失败"的方案
- ❌ 不要假设用户有充足时间，除非summary明确说明
```

### ❌ 问题2: 缺少"工具偏好记录机制"

**问题**: 今天用户明确说"我就要CLI"，但这个偏好没有记录到全局CLAUDE.md

**影响**:
- 下次遇到类似情况，AI可能又建议Web界面
- 用户需要重复表达偏好

**建议**: 增加"工具偏好"部分

```markdown
## 工具偏好

### 部署和配置
- ✅ **优先CLI**: Railway、Vercel等平台优先用CLI而非Web界面
  - 原因：CLI更快，可脚本化，可复现
  - 例外：首次配置复杂时可以用Web界面探索，然后转CLI

### 开发工具
- ✅ **优先成熟库**: 不要重复造轮子，优先用成熟的开源库
  - 例外：学习目的或库不满足需求时才自己实现

### 数据存储
- ✅ **P0阶段优先文件系统**: 快速验证时用JSON文件，不用数据库
  - 原因：零配置，易调试，易追溯
  - 例外：P1/P2阶段或数据量大时再考虑数据库

### 前端框架
- ✅ **P0阶段优先单HTML**: 快速验证时用单HTML文件，不用构建工具
  - 原因：零构建步骤，易部署
  - 例外：P1/P2阶段或交互复杂时再用React/Vue
```

### ❌ 问题3: 缺少"项目类型模板"

**问题**: 全局CLAUDE.md是通用的，但不同类型项目有不同的工作模式

**影响**:
- 黑客松项目（5小时）和长期项目（几周）的工作节奏完全不同
- AI不知道应该用什么节奏

**建议**: 增加"项目类型识别"

```markdown
## 项目类型识别

根据项目特征，自动调整工作模式：

### 🏃 黑客松/比赛模式（识别特征：有明确deadline，时间<24小时）
- **节奏**: 快速迭代，优先可演示的功能
- **质量标准**: 能跑 > 完美，先实现再优化
- **文档**: 最小化，只写评分必需的文档
- **技术选型**: 优先熟悉的技术栈，避免学习新工具
- **沟通**: 结论先行，不讨论细节，直接执行
- **错误处理**: 遇到blockers立即汇报，不要浪费时间硬试

### 🚀 快速原型模式（识别特征：验证想法，时间1-3天）
- **节奏**: 中速迭代，关注核心功能
- **质量标准**: 核心功能完整，边缘case可以忽略
- **文档**: 写README和核心设计文档
- **技术选型**: 平衡熟悉度和适配度
- **沟通**: 关键决策讨论，细节直接执行

### 🏗️ 长期项目模式（识别特征：生产环境，时间>1周）
- **节奏**: 稳健迭代，关注可维护性
- **质量标准**: 完整的错误处理、测试、文档
- **文档**: 完整的架构文档、API文档、部署文档
- **技术选型**: 优先成熟稳定的方案
- **沟通**: 充分讨论，权衡利弊

### 📚 学习项目模式（识别特征：学习新技术，无deadline压力）
- **节奏**: 慢速，关注理解原理
- **质量标准**: 代码清晰，注释详细
- **文档**: 写学习笔记，记录踩坑经验
- **技术选型**: 优先学习目标技术
- **沟通**: 详细解释原理，鼓励提问

**如何识别项目类型**：
1. 检查项目CLAUDE.md是否明确说明（如"比赛时长：5小时"）
2. 检查用户的时间表达（"要来不及了"、"deadline"、"慢慢来"）
3. 如果不确定，询问用户："这是黑客松/原型/长期项目/学习项目？"
```

### ❌ 问题4: 缺少"记忆触发器"

**问题**: 全局CLAUDE.md没有告诉AI"什么时候应该写memory"

**影响**:
- AI不知道应该记录什么
- 用户的偏好和反馈没有被持久化

**建议**: 增加"记忆触发器"

```markdown
## 记忆触发器（何时写Memory）

### 🔴 必须记录（立即写入memory）

1. **用户明确纠正AI的行为**
   - 触发词："不要..."、"别..."、"不是这样"
   - 示例："不要问我是否需要，直接做"
   - 行动：写入 `memory/feedback_*.md`

2. **用户表达强烈偏好**
   - 触发词："我就要..."、"必须..."、"一定要..."
   - 示例："我就要CLI"
   - 行动：写入 `memory/user_preferences.md`

3. **重复出现的模式（3次以上）**
   - 示例：用户连续3次选择CLI而非Web界面
   - 行动：写入 `memory/user_patterns.md`

4. **用户明确说"记住"**
   - 触发词："记住..."、"以后..."、"下次..."
   - 示例："记住，我不喜欢长篇解释"
   - 行动：根据内容类型写入对应memory

### 🟡 建议记录（任务结束后）

1. **项目特定的技术决策**
   - 示例：为什么选择A而不是B
   - 行动：写入 `memory/project_*.md`

2. **失败的尝试**
   - 示例：尝试了X但失败了，原因是Y
   - 行动：写入 `memory/lessons_*.md`

### ❌ 不要记录

1. **临时的、一次性的信息**
   - 示例："今天用这个API key"
   - 原因：会过期，不应该持久化

2. **项目代码细节**
   - 示例："X函数在Y文件的Z行"
   - 原因：代码会变化，应该从代码中读取

3. **已经在CLAUDE.md中的信息**
   - 原因：避免重复
```

### ❌ 问题5: 缺少"时间感知"

**问题**: 全局CLAUDE.md没有告诉AI如何感知时间压力

**影响**:
- 今天用户说"要来不及了"，但AI没有充分理解紧急程度
- AI的建议节奏不匹配用户的时间压力

**建议**: 增加"时间压力识别"

```markdown
## 时间压力识别

### 🔴 紧急模式（识别后立即切换）

**触发词**：
- "要来不及了"、"快"、"赶紧"、"马上"
- "deadline"、"截止"、"还有X分钟/小时"
- "急"、"紧急"、"赶时间"

**行为调整**：
1. ✅ 结论先行，不解释细节
2. ✅ 给出最快方案，不讨论多个选项
3. ✅ 直接执行，不问"是否需要"
4. ✅ 遇到blockers立即汇报，不要浪费时间尝试多种方案
5. ❌ 不要建议"完美但慢"的方案
6. ❌ 不要建议"学习新工具"
7. ❌ 不要写详细文档（除非评分必需）

**沟通模板**：
- "最快方案是X，我现在开始执行"（不是"我建议X，你觉得呢？"）
- "遇到问题Y，需要你决策：A（快但不完美）还是B（慢但完美）？"

### 🟡 正常模式

**触发词**：
- 没有明确的时间压力表达
- "慢慢来"、"不急"、"有时间"

**行为调整**：
1. ✅ 讨论多个方案，权衡利弊
2. ✅ 解释技术细节
3. ✅ 写完整文档
4. ✅ 考虑长期可维护性

### 🟢 学习模式

**触发词**：
- "为什么"、"原理"、"学习"、"理解"
- "详细解释"、"教我"

**行为调整**：
1. ✅ 详细解释原理
2. ✅ 提供多个示例
3. ✅ 鼓励提问
4. ✅ 推荐学习资源
```
