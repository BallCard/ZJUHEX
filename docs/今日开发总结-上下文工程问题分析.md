## 三、上下文工程失败的根本原因

### 1. Token限制导致的强制压缩

Claude的上下文窗口有限（200K tokens），当对话过长时，系统会自动压缩历史对话为summary。

**问题**:
- Summary只保留"主要意图"和"关键概念"，丢失了：
  - 用户的情绪状态（紧急、焦虑、不耐烦）
  - 具体的决策过程（为什么选A不选B）
  - 临时的约束条件（"这次先用CLI"、"暂时用mock数据"）
  - 上一轮对话的失败尝试（避免重复建议）

### 2. 多轮迭代中的信息衰减

从git历史看，今天有32次commit，说明经历了大量的迭代：

```
P0 (10:00-12:00) → P1 (12:00-14:00) → P2 (14:00-16:00) → 部署调试 (16:00-18:00)
```

每次上下文断裂，都会丢失一部分"为什么这样做"的背景：
- P0为什么选择单教材？（因为5小时时间约束）
- P1为什么加异步处理？（因为P0同步阻塞用户体验差）
- P2为什么替换前端？（因为Gemini设计的更好看）

### 3. 隐式知识没有显式记录

很多决策是在对话中"口头约定"的，没有写入文档：

**示例**:
- "这次先用CLI配置Railway" → 没有写入CLAUDE.md
- "用真实数据替换mock数据" → 没有写入开发日志
- "时间紧急，直接执行不讨论" → 没有写入项目文档

当上下文断裂时，这些隐式知识全部丢失。

## 五、改进建议

### 建议1: 关键决策写入CLAUDE.md

**当前问题**: 很多决策只在对话中口头约定

**改进方案**: 每次重大决策后，立即更新CLAUDE.md

**示例**:
```markdown
## 临时约束 (2026-05-10)
- 时间紧急：比赛截止18:00，优先快速修复而非完美优化
- 工具偏好：Railway配置使用CLI而非Web界面
- 数据策略：用真实解析数据替换mock数据用于demo

## 已废弃方案
- ❌ 使用Web界面配置Railway环境变量（用户偏好CLI）
- ❌ 保留旧前端（已替换为Gemini设计的新前端）
```

### 建议2: 使用Memory系统记录隐式知识

**当前问题**: 用户偏好和情绪状态没有持久化

**改进方案**: 使用Claude Code的memory功能

**示例**:

## 六、总结

### 今日上下文工程的主要问题

1. **15次上下文断裂**，导致关键信息遗漏
2. **用户情绪和紧急状态**没有被summary保留
3. **技术决策的"为什么"**在压缩中丢失
4. **临时约束和偏好**没有显式记录
5. **失败尝试**没有记录，导致AI重复建议

### 对项目的实际影响

- **时间浪费**: 用户需要重复解释需求和背景（估计浪费30-60分钟）
- **决策反复**: AI可能质疑已经确定的技术选择
- **用户体验差**: 用户感到"AI不记得之前说的话"，产生挫败感

### 核心教训

**上下文工程不是可选项，而是必需品**

在长时间、多轮迭代的开发任务中，必须主动管理上下文：
- ✅ 关键决策写入CLAUDE.md
- ✅ 用户偏好写入memory
- ✅ 临时约束用TODO标记
- ✅ 重要节点创建checkpoint文档
- ✅ 开发日志记录"为什么"而非只记录"做了什么"

**否则，每次上下文断裂都是一次"失忆"，需要用户重新教育AI。**
