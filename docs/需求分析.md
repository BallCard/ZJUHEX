# 需求分析

## 1. 问题背景

在高校教学体系中，同一学科领域往往存在多本教材并行使用，这些教材由不同出版社、不同作者编写，存在大量内容重复，导致：
- 学生大量时间浪费在重复阅读上
- 教师难以系统评估教材间的交叉与缺失
- 教学管理者缺乏数据驱动的课程整合依据

## 2. 核心目标

将7本医学教材整合压缩到不超过原始体量30%的精华版本，同时保证教学效果不打折。

## 3. 子问题分解

### 3.1 知识点粒度定义

**问题**：如何定义"知识点"的粒度？

**分析**：
- 粒度过粗：无法精确识别重复（如"炎症"整章 vs "炎症反应"小节）
- 粒度过细：增加计算复杂度，可能破坏知识连贯性

**方案**：
- **基本单位**：以"概念"为最小粒度
- **层级结构**：概念 → 知识点 → 小节 → 章节
- **提取策略**：
  - 核心概念：定义明确的术语（如"动作电位"）
  - 定理/原理：具有因果关系的陈述
  - 方法/现象：可操作的步骤或可观察的现象

**输出结构**：
```json
{
  "id": "node_001",
  "name": "动作电位",
  "definition": "细胞受到刺激后，膜电位发生的一次快速而可逆的倒转",
  "category": "核心概念",
  "chapter": "第二章 细胞的基本功能",
  "page": 35,
  "context": "周围300字上下文"
}
```

### 3.2 重复判定标准

**问题**：如何判断两个知识点是否重复？

**挑战**：
- 措辞不同："白细胞" vs "白blood细胞" vs "leukocyte"
- 描述深度不同：《生理学》简述 vs 《病理学》详述
- 角度不同：同一概念在不同教材中的应用场景不同

**方案**：双重对齐策略

**阶段1：Embedding相似度筛选**
- 模型：sentence-transformers (paraphrase-multilingual-MiniLM-L12-v2 或 BGE-small-zh)
- 阈值：0.85+（高相似度）
- 优点：快速，适合大规模筛选
- 缺点：可能误判同义词、上下位概念

**阶段2：LLM精确判断**
- 对阶段1筛选出的候选对，调用LLM判断是否等价
- Prompt设计：
  ```
  判断以下两个知识点是否描述同一概念：
  知识点A：{name_a} - {definition_a}
  知识点B：{name_b} - {definition_b}
  
  输出JSON格式：
  {
    "is_same": true/false,
    "confidence": 0.0-1.0,
    "reason": "判断理由"
  }
  ```

**判定标准**：
- `is_same=true` 且 `confidence>=0.8` → 判定为重复
- 边界情况：上下位概念（如"免疫系统"包含"T细胞"）不判定为重复，而是建立`contains`关系

### 3.3 教学连贯性保障

**问题**：整合后如何保证知识链路不断裂？

**风险**：
- 删除某个知识点后，依赖它的后续知识点失去前置基础
- 合并时选择了描述不完整的版本

**方案**：

**1. 依赖关系检查**
- 在整合前构建完整的依赖图（prerequisite关系）
- 删除节点前检查：是否有其他节点依赖它？
- 如果有依赖，必须保留或找到替代节点

**2. 合并策略**
- 优先保留描述最系统完整的版本
- 判断标准：
  - 字数更多（通常更详细）
  - 包含更多关系边（说明与其他知识点联系更紧密）
  - 来自专业性更强的教材（如"炎症"优先选《病理学》而非《生理学》）

**3. 完整性验证**
- 整合后遍历依赖图，检查是否存在"孤岛"（无前置依赖的高级概念）
- 生成"知识缺口报告"：列出可能缺失的前置知识

### 3.4 压缩比计算方式

**目标**：整合后内容≤原始总字数的30%

**计算公式**：
```
压缩比 = (整合后总字数 / 原始总字数) × 100%
```

**字数统计范围**：
- 包含：知识点定义、上下文
- 不包含：章节标题、页眉页脚、图表说明

**压缩策略**：
1. **去重**：合并重复知识点（预计减少40-50%）
2. **提纯**：删除冗余描述，保留核心定义（预计减少20-30%）
3. **优先级排序**：
   - 必须保留：核心概念、关键定理
   - 可选保留：应用案例、补充说明
   - 可删除：重复例子、过渡性描述

**动态调整**：
- 如果压缩比>30%，增加删除力度
- 如果压缩比<20%，可能过度删除，需检查教学完整性

### 3.5 RAG分块策略选择依据

**问题**：如何分块才能平衡检索精度和上下文完整性？

**分块参数**：
- **块大小**：500-800字
- **重叠窗口**：50-100字

**选择依据**：

**块大小分析**：
- 太小（<300字）：
  - 优点：检索精确
  - 缺点：上下文不足，可能截断完整概念
- 太大（>1000字）：
  - 优点：上下文丰富
  - 缺点：检索噪音大，Token消耗高
- **选择500-800字**：
  - 医学教材一个知识点通常200-500字
  - 500-800字可包含1-2个完整知识点
  - 适合LLM上下文窗口（通常8k tokens）

**重叠窗口分析**：
- 目的：防止知识点被截断
- 50-100字约占块大小的10-15%
- 示例：
  ```
  Chunk 1: [0-800字]
  Chunk 2: [750-1550字]  # 重叠50字
  ```

**元数据保留**：
每个chunk必须包含：
```json
{
  "chunk_id": "book01_ch02_chunk_003",
  "textbook": "生理学",
  "chapter": "第二章 细胞的基本功能",
  "page_start": 35,
  "page_end": 37,
  "content": "...",
  "char_count": 650
}
```

## 4. 技术约束

### 4.1 性能约束
- 单本教材解析时间：<2分钟
- 知识图谱构建时间：<5分钟/本
- RAG查询响应时间：<3秒

### 4.2 成本约束
- LLM调用：优先使用国内API（通义千问/DeepSeek），成本更低
- Embedding：优先使用本地模型（sentence-transformers），免费

### 4.3 部署约束
- 必须支持Web访问
- 部署平台：魔搭创空间（免费CPU）或其他公网平台

## 5. 验收标准

### 5.1 功能验收
- [ ] 能上传并解析7本教材
- [ ] 能构建并可视化知识图谱
- [ ] 能识别并整合重复知识点
- [ ] 压缩比≤30%
- [ ] RAG问答带引用来源
- [ ] 支持多轮对话修改整合决策

### 5.2 质量验收
- [ ] 整合决策有明确理由
- [ ] 知识图谱无孤岛节点
- [ ] RAG回答准确率>80%（基于自建benchmark）
- [ ] 引用来源准确率>90%

### 5.3 文档验收
- [ ] README可复现部署
- [ ] Agent架构说明论证充分
- [ ] 整合报告数据完整

## 6. 风险与应对

### 6.1 技术风险
- **风险**：LLM提取知识点不准确
- **应对**：使用few-shot示例，提供明确的JSON schema

### 6.2 时间风险
- **风险**：5小时内无法完成所有功能
- **应对**：优先级排序，先完成P0核心功能

### 6.3 成本风险
- **风险**：API调用费用超预算
- **应对**：使用本地模型（Ollama）或免费额度较高的API
