🔥锐评：问题与反思

1. 过度工程化倾向 ⚠️

现象：

  • P0阶段90分钟完成核心功能
  • P1/P2阶段花费6小时做"生产化"
  • 最终提交物仍是demo级别
比赛时长: 5小时 实际开发: 8小时（超时60%） 核心功能: 90分钟（18%） 优化时间: 6.5小时（82%） ← 这是问题所在

教训：5小时黑客松应该"能跑就行"，不需要异步提取器、日志系统、跨教材整合这些P1/P2特性。过早优化是万恶之源。

2. 前端反复横跳 🎭

00:00-04:00: 单HTML前端（index.html）
04:00-06:00: React前端（frontend_new/）
06:00-07:00: 部署时发现React打包问题
07:00-08:00: 回退到单HTML + 修复API路径

问题：

  • 3次前端重构，浪费2小时
  • 最终部署用的还是单HTML（回到起点）
  • React前端只用于本地开发

教训：P0的单HTML方案已经够用，不要在黑客松里追求技术栈升级。

3. 测试数据缺失 🚫

"无真实数据测试加速开发：所有服务都通过mock数据验证"
—— 开发日志原文

问题：

  • 8小时开发，0次真实PDF测试
  • 所有压缩比、RAG效果都是理论值
  • 演示时可能翻车（PDF解析失败、LLM提取质量差）

教训：没有真实数据的demo是空中楼阁。应该在Phase 0就准备好测试PDF和API密钥。

4. 文档驱动开发的陷阱 📝

文档产出：

Agent架构说明.md 开发日志.md P1实施计划.md P2实施计划.md 部署指南.md 测试指南.md 整合报告模板.md

问题：

  • 文档写得很完美，但代码没跑通
  • 最近5次commit都是修bug（API key加载、图谱加载、按钮禁用）
  • 文档说"压缩比2.30%"，但没有真实验证

教训：先让代码跑起来，再写文档。黑客松不是写PRD。

5. 部署焦虑症 🚀

部署尝试：

Railway配置 GitHub Pages ModelScope备用 Gradio界面

问题：

  • 4种部署方案，没有一个完全跑通
  • 最后还在修"前端API URL指向Railway后端"的问题
  • 部署时间占比30%

教训：黑客松demo用localhost演示就够了。部署是加分项，不是必选项。
