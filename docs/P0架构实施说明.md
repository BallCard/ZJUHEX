# P0架构实施说明

> 本文档记录P0 Demo阶段的实际实施方案，与`Agent架构说明.md`中的P1目标方案形成对照。

## 1. P0 vs P1 架构对比

| 维度 | P0实施方案 | P1目标方案 |
|------|-----------|-----------|
| **流程控制** | 顺序函数调用 | LangGraph状态机 |
| **PDF解析** | PyMuPDF (fitz) | MinerU + Docling备用 |
| **语义嵌入** | sentence-transformers (MiniLM) | BGE-M3多向量 |
| **去重策略** | 余弦相似度0.90 | MinHash 0.7 + BGE-M3 0.91 |
| **检索方式** | FAISS向量检索 | BM25+FAISS混合+Rerank |
| **压缩策略** | 节点定义字数统计 | LLMLingua动态压缩 |
| **引用验证** | 手动元数据追踪 | Judge Model验证 |
| **状态管理** | 文件系统JSON | LangGraph Checkpoint |
| **可视化** | 无（跳过Phase 7） | Cytoscape.js (P0) / AntV G6 (P1) |
| **开发时间** | ~90分钟 | 预计4-5小时 |

## 2. P0架构设计

### 2.1 整体架构

P0采用**顺序流水线架构**，通过FastAPI端点串联各个服务模块：

```
用户上传PDF
    ↓
[POST /api/upload] 保存文件，生成job_id
    ↓
[POST /api/parse/{job_id}] parser.py: PyMuPDF解析
    ↓
[POST /api/build_graph/{job_id}] knowledge_graph.py: LLM提取（前10块）
    ↓
[POST /api/integrate/{job_id}] integration.py: 语义去重（0.90阈值）
    ↓
[POST /api/generate_report/{job_id}] report_generator.py: 计算压缩比
    ↓
[POST /api/rag/index/{job_id}] rag.py: FAISS索引
    ↓
[POST /api/rag/query/{job_id}] rag.py: 检索+生成回答
```

### 2.2 为什么选择顺序流水线？

**P0目标**: 5小时内完成单教材端到端演示

**顺序流水线优势**:
1. **开发速度快**: 无需学习LangGraph API，直接函数调用
2. **调试简单**: 每个步骤独立测试，错误定位清晰
3. **状态透明**: 文件系统JSON持久化，所有中间结果可见
4. **满足Demo需求**: P0只需证明技术可行性，不需要生产级容错

**权衡**:
- ❌ 无断点恢复（失败需从头开始）
- ❌ 无动态调整（压缩比不达标无法自动回退）
- ❌ 无并行处理（所有步骤串行执行）
- ✅ 但满足P0 Demo范围（单教材前10块）

### 2.3 核心服务模块

#### 2.3.1 parser.py - 文档解析

**技术选型**: PyMuPDF (fitz)

**选择理由**:
- ✅ 纯Python实现，无外部依赖
- ✅ 安装简单（`pip install PyMuPDF`）
- ✅ 中文支持良好
- ✅ 提取速度快（~1秒/页）

**P1升级方向**: MinerU专为中文医学PDF优化，但需要额外配置

**实现细节**:
```python
# 分块策略: 500-800字符
chunk_size_min = 500
chunk_size_max = 800

# 段落分割: 双换行符或句号+换行
paragraphs = re.split(r'\n\n+|\n(?=[。！？])', text)

# 长段落按句子切分
sentences = re.split(r'([。！？])', paragraph)
```

#### 2.3.2 knowledge_graph.py - 知识图谱构建

**技术选型**: DeepSeek API + 自定义JSON结构

**LLM Prompt设计**:
```python
prompt = f"""你是医学知识图谱构建专家。从以下文本中提取知识点和关系。

文本内容：
{chunk['content']}

请提取：
1. 核心概念（名词、术语、定义）
2. 概念之间的关系（前置依赖、并列、包含、应用）

以JSON格式返回，格式如下：
{{
    "concepts": [
        {{"name": "概念名称", "type": "concept", "definition": "简短定义"}},
        ...
    ],
    "relationships": [
        {{"source": "概念A", "target": "概念B", "type": "前置依赖|并列|包含|应用", "description": "关系说明"}},
        ...
    ]
}}

只返回JSON，不要其他文字。"""
```

**P0限制**: 只处理前10个chunk（时间约束）

**P1升级方向**: LlamaIndex PropertyGraphIndex自动构建

#### 2.3.3 integration.py - 语义去重

**技术选型**: sentence-transformers + 余弦相似度

**模型**: `paraphrase-multilingual-MiniLM-L12-v2`
- 384维向量
- 支持中文
- 模型大小~120MB

**去重算法**:
```python
# 1. 计算所有节点的嵌入
embeddings = model.encode(node_texts)

# 2. 计算余弦相似度矩阵
similarity_matrix = cosine_similarity(embeddings)

# 3. 聚类合并（阈值0.90）
for i, j in pairs:
    if similarity_matrix[i, j] >= 0.90:
        merge_nodes(i, j)
```

**阈值选择**: 0.90（生物医学领域最佳实践）
- 0.85: 过度合并，可能误删不同概念
- 0.90: 平衡精度和召回
- 0.95: 过于保守，重复概念无法去除

**P1升级方向**: MinHash预聚类 + BGE-M3精确判断

#### 2.3.4 report_generator.py - 报告生成

**压缩比计算**:
```python
# 原始字符数
original_chars = sum(chunk["char_count"] for chunk in chunks)

# 整合后字符数（节点定义+标签）
deduplicated_chars = sum(
    len(node["definition"]) + len(node["label"])
    for node in deduplicated_graph["nodes"]
)

# 压缩比
compression_ratio = (deduplicated_chars / original_chars) * 100%
```

**报告内容**:
1. 压缩比统计（是否≤30%）
2. 知识图谱统计（节点数、边数）
3. 整合决策摘要（去重策略、阈值）
4. 整合后知识内容（所有节点列表）
5. 知识关系网络（所有边列表）
6. 教学完整性评估
7. 典型案例（合并节点示例）

**P1升级方向**: LLMLingua动态压缩（保留完整图谱，检索时压缩）

#### 2.3.5 rag.py - RAG问答

**技术选型**: FAISS + sentence-transformers + DeepSeek

**检索流程**:
```python
# 1. 生成查询向量
query_embedding = model.encode([question])

# 2. FAISS检索top-3
distances, indices = faiss_index.search(query_embedding, k=3)

# 3. 构建上下文
context = "\n\n".join([
    f"[来源: {chunk['textbook']}, 第{chunk['page']}页]\n{chunk['content']}"
    for chunk in retrieved_chunks
])

# 4. LLM生成回答
answer = llm.complete(f"上下文：{context}\n问题：{question}")
```

**引用追踪**: 手动元数据
```python
citations = [
    {
        "textbook": chunk["textbook"],
        "page": chunk["page"],
        "content": chunk["content"][:200] + "...",
        "chunk_id": chunk["chunk_id"]
    }
    for chunk in retrieved_chunks
]
```

**P1升级方向**: BM25+FAISS混合检索 + bge-reranker重排 + Judge Model验证

### 2.4 状态管理

**P0方案**: 文件系统JSON

```
data/runtime/jobs/{job_id}/
├── state.json              # 任务状态
├── parsed_chunks.json      # 解析结果
├── knowledge_graph.json    # 原始图谱
├── deduplicated_graph.json # 去重后图谱
├── faiss.index             # FAISS索引
└── chunks_for_rag.json     # RAG分块
```

**state.json结构**:
```json
{
  "job_id": "abc123",
  "status": "completed",
  "current_phase": "rag_indexed",
  "filename": "03_生理学.pdf",
  "chunks_count": 150,
  "nodes_count": 45,
  "compression_ratio": 28.5,
  "created_at": "2026-05-10T12:00:00",
  "updated_at": "2026-05-10T12:15:00"
}
```

**优势**:
- ✅ 零配置（无需数据库）
- ✅ 可追溯（所有中间结果可查看）
- ✅ 易调试（直接打开JSON文件）

**P1升级方向**: LangGraph Checkpoint（支持断点恢复）

## 3. 前端设计

**技术选型**: 单HTML文件

**选择理由**:
- ✅ 无构建步骤（无需npm/webpack）
- ✅ 部署简单（直接打开浏览器）
- ✅ 满足P0 Demo需求

**核心功能**:
1. 文件上传（拖拽式界面）
2. 流水线执行（进度条可视化）
3. 统计展示（压缩比、节点数）
4. 报告查看（Markdown渲染）
5. RAG问答（带引用来源）

**P1升级方向**: React + Ant Design + Cytoscape.js可视化

## 5. 已知限制与风险

### 5.1 未经真实数据验证

**风险**: 所有服务都通过mock数据测试，真实场景可能出现：
- PDF解析失败（特殊格式、扫描版）
- LLM提取质量差（医学术语识别错误）
- 去重阈值不合适（过度合并或合并不足）
- 压缩比无法达标（>30%）

**缓解**: 需要用真实PDF和API密钥进行端到端测试

### 5.2 无错误恢复机制

**风险**: 流水线任意步骤失败，需从头开始

**P1解决方案**: LangGraph Checkpoint支持断点恢复

### 5.3 压缩比计算不完整

**当前**: 只计算节点定义+标签字数

**问题**: 未包含关系描述、教学说明等内容

**P1解决方案**: LLMLingua动态压缩完整图谱内容

### 5.4 单教材Demo范围

**当前**: 只处理单教材前10个chunk

**P1目标**: 7本教材完整处理

## 6. P0→P1升级路径

### 6.1 短期优化（1-2天）

1. **端到端测试**: 用真实PDF和API密钥验证完整流程
2. **错误处理**: 添加try-catch和重试机制
3. **压缩比优化**: 调整去重阈值，确保≤30%
4. **可视化**: 集成Cytoscape.js展示知识图谱

### 6.2 中期升级（1周）

1. **LangGraph迁移**: 顺序流水线 → 状态机
2. **BGE-M3升级**: sentence-transformers → BGE-M3多向量
3. **混合检索**: FAISS → BM25+FAISS+Rerank
4. **7本教材**: 单教材 → 跨教材整合

### 6.3 长期优化（2-4周）

1. **MinerU解析**: PyMuPDF → MinerU（中文医学PDF优化）
2. **Judge Model**: 手动追踪 → 自动验证引用
3. **LLMLingua压缩**: 静态压缩 → 动态压缩
4. **AntV G6**: Cytoscape.js → G6（WebGL性能优化）
5. **CMeKG对齐**: 自建图谱 → 与中文医学知识图谱对齐

## 7. 总结

P0实施方案采用**顺序流水线架构**，在90分钟内完成单教材端到端演示，验证了核心技术可行性：
- ✅ PDF解析与分块
- ✅ LLM知识提取
- ✅ 语义去重（0.90阈值）
- ✅ 压缩比计算
- ✅ RAG问答（带引用）

**核心权衡**: 牺牲容错性、扩展性、性能优化，换取快速开发和Demo可行性验证。

**下一步**: 配置API密钥 → 端到端测试 → 根据测试结果调整参数 → 准备演示材料。
